# Feature Research: v1.1 Branding, Docs, and Hardening

**Domain:** CLI tool publishing, documentation, CI/CD, release automation, UX presets
**Researched:** 2026-07-16
**Confidence:** HIGH (based on webreel reference implementation, Bun/Biome/Zod README patterns, Mintlify docs structure, changesets ecosystem, and ffmpeg testing patterns)

## Feature Landscape

### Table Stakes (Users Expect These)

Features users assume exist for any published open-source CLI tool. Missing these = product feels incomplete or unprofessional.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| README.md with logo, install, quickstart | First thing users see on GitHub/npm. No README = invisible project. | LOW | See README section below for structure |
| LICENSE file (Apache 2.0) | Already decided. Users check license before adopting. | LOW | Single file, update `license` field in both package.json files |
| CONTRIBUTING.md | Contributors need to know how to set up, test, and submit PRs | LOW | Dev setup, testing, PR process |
| CHANGELOG.md | Users need to know what changed between versions. Required for trust. | LOW | Auto-generated by changesets |
| npm publishing | `npx tuireel` and `bunx tuireel` must work. Not published yet. | MEDIUM | Needs `publishConfig`, npm token, correct `files` field |
| GitHub releases with tags | Standard distribution channel. Users expect semver tags + release notes. | LOW | Automated by changesets GitHub Action |
| Basic CI (lint, type-check, build, test) | PRs without CI = untrustworthy. Table stakes for any OSS project. | LOW | webreel pattern: pnpm/action-setup + actions/setup-node + matrix |
| Error messages with actionable guidance | "ffmpeg not found" is bad. "ffmpeg not found. Run `tuireel record` to auto-download, or install manually: https://..." is good. | MEDIUM | Audit all throw sites in core |
| Graceful signal handling | Ctrl+C during long recording must not leave zombie processes or corrupt output | MEDIUM | Already partially implemented in recorder.ts, needs hardening |

### Differentiators (Competitive Advantage)

Features that set tuireel apart from VHS, asciinema, terminalizer. Not required but valuable.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Mintlify documentation site | Professional docs with search, guides, API reference. VHS has a basic site; tuireel can do much better. | MEDIUM | `docs/` folder in monorepo, `docs.json` config, MDX pages |
| User-facing presets | `"preset": "polished"` in config gives cursor + HUD + sound without manual config. Lowers barrier to polished output. | LOW | Named bundles mapping to existing config fields: theme, sound, cursor, HUD |
| Full branding package | Logo, color palette, banner, social card. Makes project feel established and maintained. | MEDIUM | SVG logo, OG image, terminal screenshot banner |
| Performance profiling + optimization | Large recordings (1000+ frames) should composite efficiently. Bottleneck is Sharp per-frame compositing. | HIGH | Measure first. Batch compositing, worker threads, skip unchanged frames |
| Release automation (changesets) | One-command release flow: changeset -> version PR -> auto-publish. Removes friction for ongoing maintenance. | MEDIUM | changesets + GitHub Action, following webreel pattern exactly |
| CI regression for video output | Catch encoding regressions across MP4/WebM/GIF. No competitor does this well. | HIGH | See CI section below for patterns |
| Diagnostics mode (`--verbose`, `--debug`) | When things go wrong, users need to self-diagnose. ffmpeg stderr, tuistory errors, frame counts. | LOW | Structured logging, `--verbose` flag, error context enrichment |

### Anti-Features (Commonly Requested, Often Problematic)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Full Mintlify API playground | "Document the programmatic API" | `@tuireel/core` is not a REST API; API playground is for HTTP APIs. Adds complexity for minimal value. | TypeDoc-style reference page in Mintlify, or just good examples |
| Interactive config builder (web UI) | "Let users build configs visually" | Huge scope, maintenance burden, splits attention from CLI. Would need its own deployment. | JSON Schema + IDE autocompletion (already shipped) is the right answer |
| Automatic changelog from commits | "Just parse conventional commits" | Changesets is better for monorepos because it lets you describe changes from the user perspective, not the commit perspective | Use changesets, not conventional-commits |
| Docker-based CI for video testing | "Run ffmpeg in Docker for reproducibility" | Adds container overhead, slows CI, Docker-in-Docker complexity. ffmpeg is deterministic enough with version pinning. | Install ffmpeg via apt in GitHub Actions runner directly |
| Mintlify versioned docs | "Support multiple doc versions for different releases" | Premature for v1.1. Adds complexity. Only needed when breaking changes happen frequently. | Single version docs, use changelog for version history |
| Custom preset authoring (user-defined presets) | "Let users create and share presets" | Requires preset file format, discovery, validation. Premature for v1.1. | Ship built-in presets only. Users can copy config manually. |


## Deep Dives by Feature Area

### 1. README Structure

Best-in-class CLI tool READMEs follow a consistent structure. Reference: Bun, Biome, Zod, changesets.

**Required sections (in order):**

1. **Banner/Logo** -- Centered image (`<p align="center">`), links to docs/Discord/issues
2. **One-line description** -- What it is in one sentence
3. **Badges** -- npm version, CI status, license, Discord/community link
4. **What is [tool]?** -- 2-3 paragraph explanation with a code example showing the happy path
5. **Install** -- All installation methods (npm, pnpm, bun, curl) with platform notes
6. **Quick start** -- Minimal working example from zero to output
7. **Documentation links** -- Link to full docs site, organized by topic
8. **Contributing** -- Link to CONTRIBUTING.md
9. **License** -- One line + link

**Key patterns observed:**
- Bun: Logo + badges + install methods for every platform + massive quick links section to docs
- Biome: Logo + badges + "What is Biome?" + install + quick links + sponsors
- changesets: Banner image + badges + conceptual explanation + how-to links + "Cool projects using" section
- All use `<p align="center">` for logo/badges alignment

**Tuireel-specific additions:**
- Terminal screenshot/GIF showing tuireel producing a demo video (dogfooding!)
- Before/after: "manual recording" vs "tuireel config"
- Quick link to presets for instant polished output

**Complexity:** LOW. Template-driven, most content references existing features.
**Dependencies:** Logo (branding), docs site URL (Mintlify), npm package published.

### 2. Mintlify Documentation Site

Mintlify uses a `docs/` folder with MDX files and a `docs.json` configuration file. Deployed via GitHub App (auto-deploy on push to main).

**Minimum viable docs site (7-10 pages):**

```
docs/
  docs.json              # Navigation, theme, name, logo, colors, favicon
  index.mdx              # Introduction / landing page
  quickstart.mdx         # Install + first recording in 5 minutes
  configuration/
    overview.mdx          # Config file format, JSON Schema reference
    steps.mdx             # All step types with examples
    themes.mdx            # Built-in themes, custom themes
    presets.mdx           # Preset system (new in v1.1)
    multi-video.mdx       # Multi-video configs, includes
    sound.mdx             # Sound effects, custom audio tracks
  cli/
    overview.mdx          # All CLI commands summary
    record.mdx            # tuireel record
    composite.mdx         # tuireel composite
    preview.mdx           # tuireel preview
    validate.mdx          # tuireel validate
    init.mdx              # tuireel init
  guides/
    first-demo.mdx        # End-to-end walkthrough: config to published GIF
    github-readme.mdx     # Embed tuireel output in GitHub README
    ci-automation.mdx     # Run tuireel in CI to auto-generate demos
  changelog.mdx           # Auto-generated or linked from CHANGELOG.md
```

**docs.json structure:**
```jsonc
{
  "name": "Tuireel",
  "logo": { "dark": "/logo/dark.svg", "light": "/logo/light.svg" },
  "favicon": "/favicon.svg",
  "colors": { "primary": "#...", "light": "#...", "dark": "#..." },
  "navigation": [
    { "group": "Getting Started", "pages": ["index", "quickstart"] },
    { "group": "Configuration", "pages": ["configuration/overview", "configuration/steps", ...] },
    { "group": "CLI Reference", "pages": ["cli/overview", "cli/record", ...] },
    { "group": "Guides", "pages": ["guides/first-demo", ...] }
  ],
  "footerSocials": { "github": "https://github.com/tuireel/tuireel" }
}
```

**Monorepo setup:** Set `docs/` as the docs root in Mintlify dashboard. Mintlify GitHub App watches this folder only. No build step needed; Mintlify builds MDX server-side.

**Complexity:** MEDIUM. Content authoring is the main effort; Mintlify infra is straightforward.
**Dependencies:** Branding (logo, colors), published npm package (for install instructions), at least one working example recording.

### 3. CI for Video/Media Output

Testing video output in CI is harder than testing code. The challenge: video files are binary, ffmpeg output varies slightly across versions, and frame-by-frame comparison is expensive.

**Practical CI strategy (tiered):**

**Tier 1: Smoke tests (must have, LOW complexity)**
- Run `tuireel record` with a minimal config in CI
- Assert output file exists, has non-zero size, correct extension
- Assert ffmpeg probe shows expected codec, resolution, duration range, fps
- Tool: `ffprobe -v quiet -print_format json -show_format -show_streams output.mp4`
- Run for all 3 formats: MP4, WebM, GIF

**Tier 2: Structural validation (should have, MEDIUM complexity)**
- Extract frame count: `ffprobe -v error -count_frames -show_entries stream=nb_read_frames`
- Assert frame count is within expected range (not 0, not wildly off)
- Assert audio stream exists when sound is configured
- Assert no audio stream when sound is not configured
- For GIF: assert frame dimensions match config

**Tier 3: Visual regression (nice to have, HIGH complexity)**
- Extract key frames at known timestamps
- Compare against reference screenshots using perceptual hash (pHash)
- Tools: `sharp` to extract frames, `imghash` or `blockhash` npm packages for perceptual comparison
- Threshold-based: allow small differences (ffmpeg version variance)
- Store reference images in repo as test fixtures
- NOT pixel-perfect comparison (too brittle across ffmpeg versions)

**GitHub Actions specifics:**
- ffmpeg: pre-installed on `ubuntu-latest` runners (ffmpeg 4.4+), or use `apt-get install ffmpeg` for specific version
- tuistory dependency: needs a display server; may need `xvfb-run` or headless mode
- Matrix: test on ubuntu-latest (primary) only for v1.1; macOS later
- Cache: cache `~/.tuireel` ffmpeg downloads between runs
- Artifacts: upload produced videos as GitHub Actions artifacts for manual inspection on failures

**CI workflow structure:**
```yaml
jobs:
  lint-and-typecheck:
    # Fast: lint, format, type-check, build
  unit-tests:
    # Fast: vitest unit tests (no video)
  integration-tests:
    # Slow: actual recording + output validation
    # Needs ffmpeg, possibly xvfb
    # Produces video artifacts
```

**Reference:** webreel CI runs lint + format + type-check + build + test in a single job. For v1.1, splitting fast/slow jobs is a differentiator.

**Complexity:** Tier 1 = LOW, Tier 2 = MEDIUM, Tier 3 = HIGH.
**Dependencies:** Working record pipeline, ffmpeg available in CI, test fixture configs.

### 4. Release Automation (pnpm Monorepo)

**Recommended: Changesets** (not semantic-release). Changesets is the standard for pnpm monorepos. Used by Astro, Biome, SvelteKit, Remix, Clerk, wagmi, webreel.

**Why changesets over semantic-release:**
- Designed for monorepos with workspace protocol (`workspace:*`)
- Human-written changelogs (not auto-generated from commits)
- Versioning PR workflow: bot opens a PR that bumps versions + updates changelogs
- `pnpm publish` resolves `workspace:*` to real versions (npm publish does NOT)

**Setup:**
1. `pnpm add -Dw @changesets/cli`
2. `pnpm changeset init` (creates `.changeset/config.json`)
3. Configure fixed versioning group: `["@tuireel/core", "tuireel"]` (both bump together)
4. Add `"access": "public"` for scoped package
5. Add GitHub Action for automated versioning PRs and publishing

**.changeset/config.json:**
```jsonc
{
  "$schema": "https://unpkg.com/@changesets/config@3.1.2/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [["@tuireel/core", "tuireel"]],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "privatePackages": { "version": true, "tag": false }
}
```

**GitHub Action (release.yml):**
```yaml
name: Release
on:
  push:
    branches: [main]
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: pnpm }
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - uses: changesets/action@v1
        with:
          publish: pnpm changeset publish
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

**Workflow:**
1. Developer runs `pnpm changeset` locally, selects packages, writes summary
2. Commits `.changeset/*.md` file with PR
3. On merge to main, GitHub Action opens "Version Packages" PR
4. Merging that PR triggers auto-publish to npm + GitHub release creation

**Package.json updates needed:**
- Both packages: `"license": "Apache-2.0"` (currently UNLICENSED)
- `@tuireel/core`: needs `"publishConfig": { "access": "public" }`
- Root: add `"changeset": "changeset"`, `"release": "changeset publish"` scripts

**Complexity:** MEDIUM. Mostly config, but needs npm token setup and workflow testing.
**Dependencies:** npm account/org, GitHub secrets, correct package.json `files` field.

### 5. Presets Pattern

Presets are named configuration bundles. The user references a name; the tool expands it to full configuration.

**How media tools do presets:**
- **ffmpeg:** `-preset ultrafast|superfast|veryfast|faster|fast|medium|slow|slower|veryslow` -- named speed/quality tradeoffs
- **ImageMagick:** Named color profiles and resize strategies
- **Sharp:** Preset options for JPEG/PNG/WebP output quality
- **Tailwind CSS:** Theme presets that extend/override default config

**Tuireel preset design:**

A preset maps to existing config fields: `theme`, `sound`, `cursor` (CursorConfig), `hud` (HudConfig).

```jsonc
// User config with preset
{
  "preset": "polished",
  "output": "demo.mp4",
  "steps": [...]
}

// Equivalent expanded config
{
  "theme": "catppuccin",
  "sound": { "effects": { "click": 1, "key": 1 }, "effectsVolume": 0.4 },
  // cursor and HUD use defaults (already enabled)
  "output": "demo.mp4",
  "steps": [...]
}
```

**Built-in presets for v1.1:**

| Preset | Theme | Sound | Cursor | HUD | Use Case |
|--------|-------|-------|--------|-----|----------|
| `polished` | catppuccin | click+key effects | visible, animated | visible, bottom | Default "looks great" option |
| `minimal` | tokyo-night | none | visible, animated | hidden | Clean, no distractions |
| `demo` | dracula | click+key effects | visible | visible, bottom | Conference talks, README GIFs |
| `silent` | (none/default) | none | hidden | hidden | Raw terminal capture, CI |

**Implementation pattern:**
```typescript
// packages/core/src/presets/built-in.ts
export const builtInPresets = {
  polished: { theme: "catppuccin", sound: { effects: { click: 1, key: 1 }, effectsVolume: 0.4 } },
  minimal: { theme: "tokyo-night" },
  demo: { theme: "dracula", sound: { effects: { click: 2, key: 2 }, effectsVolume: 0.5 } },
  silent: { /* cursor: { visible: false } */ },
} as const;
```

**Resolution order:** preset defaults < config file values (user overrides win).

**Schema change:** Add optional `preset` field to config schema. Validate preset name against known names. Apply preset values as defaults before user config.

**Complexity:** LOW. It is a thin layer over existing config fields.
**Dependencies:** Existing config schema, theme system, sound system, overlay types.

### 6. Branding

Successful OSS CLI tools invest in visual identity. It signals maintenance, seriousness, and community.

**Minimum branding set:**

| Element | Purpose | Where Used | Format |
|---------|---------|------------|--------|
| Logo (icon) | Primary brand mark | README, docs, npm, favicon | SVG + PNG (128px, 256px, 512px) |
| Logo (wordmark) | Name + icon | README header, docs header | SVG |
| Color palette | Visual consistency | Docs theme, logo, social cards, terminal output | 3-5 colors: primary, secondary, accent, bg-dark, bg-light |
| Banner image | README hero, social sharing | GitHub repo, README | PNG 1280x640 (GitHub OG size) |
| Social card (OG image) | Link previews on Twitter/Slack/Discord | GitHub repo settings, docs meta | PNG 1200x630 |
| Favicon | Browser tab, docs site | Mintlify docs | SVG or ICO |

**Color palette approach:**
- Primary color: something that reads well on both dark and light backgrounds
- Terminal-inspired: use the tool's own theme colors (catppuccin or similar) as brand palette base
- Reference: Biome uses orange/coral, Bun uses peach/salmon, Deno uses dark teal, Warp uses purple

**Logo considerations:**
- Terminal/screen iconography (terminal window, film reel, play button)
- Must work at 16px favicon size
- Must work in dark mode and light mode
- SVG source, export PNGs at standard sizes

**Banner image:**
- Show a terminal with tuireel config on left, produced video output on right
- Or: terminal screenshot with the tuireel HUD/cursor overlay visible
- Dogfooding: use tuireel itself to generate the demo GIF for the README

**Complexity:** MEDIUM. Design work is the bottleneck, not code.
**Dependencies:** None (can be done in parallel with everything else).

### Dependency Notes

- **README requires Branding:** Logo and screenshot/GIF are the README hero. Do branding first or in parallel.
- **Mintlify requires Branding:** docs.json needs logo paths and color hex values.
- **npm publish requires Changesets + package.json updates:** License field, publishConfig, correct files field.
- **Integration tests require ffmpeg in CI:** GitHub Actions ubuntu-latest has ffmpeg pre-installed, but version may vary.
- **Presets are independent:** Can be built and shipped without any other v1.1 feature. Low risk.
- **Diagnostics are independent:** Can be built incrementally. Start with --verbose flag, enrich error messages over time.

## MVP Definition

### Launch With (v1.1)

Minimum set to ship tuireel as a credible, findable, installable open-source project.

- [ ] **README.md** -- Logo, badges, install, quickstart, feature overview, docs link
- [ ] **LICENSE** (Apache 2.0) -- Single file + package.json license field updates
- [ ] **CONTRIBUTING.md** -- Dev setup, testing, PR expectations
- [ ] **Changesets + release automation** -- .changeset config, GitHub Action, npm token
- [ ] **First npm publish** -- Both packages published, `npx tuireel` works
- [ ] **Basic CI** (lint, type-check, build, unit tests) -- GitHub Actions workflow
- [ ] **Mintlify docs site** (5-7 core pages) -- Introduction, quickstart, config overview, steps reference, CLI reference
- [ ] **Presets** (3-4 built-in) -- polished, minimal, demo, silent
- [ ] **Branding** (logo + colors + favicon) -- Minimum viable visual identity
- [ ] **Diagnostics** (--verbose flag + better error messages) -- Actionable ffmpeg/tuistory errors

### Add After Validation (v1.1.x)

Features to add once core v1.1 is working and feedback comes in.

- [ ] **CI integration tests** (Tier 1: smoke tests) -- Actual recording + output validation in CI
- [ ] **Social card / OG image** -- For GitHub repo and docs site link previews
- [ ] **Banner image** -- README hero showing tuireel in action
- [ ] **Performance profiling** -- Identify and fix compositor bottlenecks for large recordings
- [ ] **CHANGELOG.md page in docs** -- Auto-linked from changesets output
- [ ] **More docs pages** -- Guides (first-demo, github-readme, ci-automation)
- [ ] **CONTRIBUTING.md linked from docs** -- Contributor guide in Mintlify

### Future Consideration (v2+)

- [ ] **CI visual regression (Tier 3)** -- Perceptual hash comparison of key frames. Defer until output stability is proven.
- [ ] **Custom user presets** -- User-defined preset files. Defer until preset UX is validated.
- [ ] **Mintlify API reference** -- TypeDoc-style reference for @tuireel/core programmatic API. Defer until API stabilizes.
- [ ] **macOS CI runner** -- Test on macOS in addition to Linux. Defer until user demand.
- [ ] **Versioned docs** -- Multiple doc versions for breaking changes. Defer until v2.0.

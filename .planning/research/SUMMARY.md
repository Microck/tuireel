# Research Summary: Tuireel v1.1 — Branding, Docs & Hardening

## One-Line Summary

v1.1 transforms tuireel from a working-but-unpublished monorepo into a polished open-source project with full branding, Mintlify docs, GitHub Actions CI, automated npm releases, reliability hardening, and user-facing presets.

## Stack Additions

| Area | New Tool | Version | Why |
|------|----------|---------|-----|
| Documentation | Mintlify (`mint` CLI) | v4.2.x | Hosted docs, `docs/` folder, `docs.json` config |
| Linting | ESLint + typescript-eslint + Prettier | 10.x / 8.x / 3.8.x | Matches webreel setup, flat config |
| Release | @changesets/cli + changesets/action | 2.30.x / v1 | Monorepo versioning, auto-CHANGELOG, npm publish |
| CI | GitHub Actions (setup-ffmpeg, pnpm/action-setup) | v3 / v4 | Two-job CI: fast lint/build/test + video regression |
| Profiling | `node --prof` + `hyperfine` | built-in | No npm packages needed |
| Branding | Sharp (existing) + `favicons` | 7.2.x | SVG-to-PNG, favicon generation |

**Not adding:** Docusaurus/VitePress (Mintlify is hosted), semantic-release (changesets better for pnpm), Biome (diverges from webreel reference), Docker-based CI (overkill).

## Feature Priorities

### Table Stakes (must ship)
- README.md with logo, install, quickstart
- LICENSE file (Apache 2.0)
- CONTRIBUTING.md
- CHANGELOG.md (auto-generated by changesets)
- npm publishing (`npx tuireel`, `bunx tuireel`)
- GitHub releases with tags
- Basic CI (lint, type-check, build, test)
- Error messages with actionable guidance
- Graceful signal handling (Ctrl+C cleanup)

### Differentiators (high value)
- Mintlify docs site (guides, config reference, CLI reference, examples)
- User-facing presets (`"preset": "polished"` for instant quality)
- Full branding package (logo, color palette, banner, OG image)
- Diagnostics mode (`--verbose`, `--debug`)
- CI regression for video output (MP4/WebM/GIF matrix)
- Release automation via changesets

### Anti-Features (explicitly excluded)
- Mintlify API playground (not a REST API)
- Interactive config builder (JSON Schema is enough)
- Docker-based CI for video testing (overkill)
- Mintlify versioned docs (premature)
- Custom user-defined presets (premature)

## Architecture Summary

**New directory structure adds:**
- `docs/` — Mintlify site (NOT a pnpm workspace member)
- `assets/branding/` — Logo, banner, OG image
- `.github/workflows/` — CI, release, video regression
- `.changeset/` — Changesets config

**Key integration points:**
- Presets: `packages/core/config/presets.ts` → config resolver merges defaults
- Reliability: AbortController + timeout guards in executor, encoder, compositor
- Diagnostics: `--verbose`/`--debug` flags → structured logger in core
- Release: `workspace:*` rewritten to exact versions by changesets at publish

**Build order (dependency-driven):**
1. Branding + repo polish (LICENSE, README, CONTRIBUTING) — no code deps
2. Presets + reliability hardening — core changes
3. Mintlify docs — needs branding + presets defined
4. CI + linting — validates everything
5. Release automation — needs CI passing
6. Performance profiling — needs stable pipeline

## Top Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| `workspace:*` not rewritten on publish | npm install breaks for users | Changesets handles this; verify in dry-run |
| Sharp native binary in CI | Build fails on Linux runners | Use `--platform=linux` in pnpm install |
| ffmpeg not available in CI | Video tests fail | `setup-ffmpeg@v3` action, cache the binary |
| Non-deterministic video output | Flaky regression tests | Compare metadata (duration, codec, resolution), not pixels |
| Mintlify `docs.json` misconfigured | Docs don't deploy | Use `mint dev` locally to validate before push |
| UNLICENSED blocks changesets | Changesets silently skips packages | Change license to Apache-2.0 BEFORE changesets setup |

## Key Insight

The v1.0 pipeline works. v1.1 is not about building features — it's about **packaging, polish, and confidence**. The highest-risk items are the CI video regression (ffmpeg in GitHub Actions) and release automation (`workspace:*` dependency rewriting). Everything else is template-driven and well-documented.

Branding must come first because README, docs, and npm all reference it. License must be set early because changesets depends on it.

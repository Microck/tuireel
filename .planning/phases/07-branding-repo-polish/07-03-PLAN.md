  <action>
Create the `.github/ISSUE_TEMPLATE/` directory structure.

1. **bug-report.yml** — Structured bug report form. Use YAML format (NOT markdown). Include:
   - `name: Bug report`
   - `description: Report an issue with Tuireel`
   - `labels: [bug]`
   - Body fields:
     - `textarea` (required): "Describe the bug" — what happened vs what was expected
     - `textarea` (required): "Config to reproduce" — paste `.tuireel.jsonc` config, render as `jsonc`
     - `textarea` (required): "System info" — OS, Node version, tuireel version, ffmpeg version, render as `shell`
     - `textarea` (optional): "Error output" — stderr/error messages, render as `shell`
   - Each field needs `id`, `attributes.label`, `attributes.description`, and `validations.required` where applicable

2. **feature-request.yml** — Feature request form:
   - `name: Feature request`
   - `description: Suggest an idea for Tuireel`
   - `labels: [enhancement]`
   - Body fields:
     - `textarea` (required): "Description" — problem or use case, placeholder: "As a Tuireel user I want [goal] so that [benefit]."
     - `textarea` (required): "Suggested solution" — how it could work
     - `textarea` (optional): "Alternatives considered"

3. **config.yml** — Template chooser config:
   - `blank_issues_enabled: false` — prevents unstructured blank issues

Use the research examples as a reference but ensure YAML is syntactically valid (proper indentation, no tabs, all required top-level fields present: `name`, `description`, `body`).
  </action>

Create `.github/pull_request_template.md` with three sections:

```markdown
## What

<!-- Brief description of what this PR does -->

## Why

<!-- Why is this change needed? Link to issue if applicable -->

## How to test

<!-- Steps for reviewer to verify the change -->
```

Keep it minimal — just these three sections with HTML comment placeholders. Don't add checklists, checkboxes, or lengthy instructions. The goal is to prompt contributors to explain their change, not to create paperwork.

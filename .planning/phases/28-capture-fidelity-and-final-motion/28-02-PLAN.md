```typescript
function resolveTerminalFrameFileIndex(
  terminalFrames: number[],
  frameIndex: number,
  maxIndex: number,
): number {
  if (maxIndex <= 0) {
    return 0;
  }
  if (terminalFrames.length === 0) {
    return Math.min(frameIndex, maxIndex);
  }
  let selectedIndex = 0;
  for (let index = 0; index < terminalFrames.length; index += 1) {
    const terminalFrameIndex = terminalFrames[index];
    if (terminalFrameIndex === undefined) {
      continue;
    }
    if (terminalFrameIndex > frameIndex) {
      break;
    }
    selectedIndex = index;
  }
  return clamp(selectedIndex, 0, maxIndex);
}
```

const outputFrameCount = Math.max(effectiveFrames.length, 1);

**Rationale:** `effectiveFrames.length` is the timeline frame count at output fps — it represents the authored wall-clock timing. The decoded frame files (`effectiveFrameFiles`) are a **pool** to sample from via `resolveTerminalFrameFileIndex`, not a sequence to play linearly. With denser capture from CAP-02, `effectiveFrameFiles.length` may exceed `effectiveFrames.length`, and using `Math.max` would stretch the video beyond its authored duration.

**Edge case guard:** If `effectiveFrameFiles.length` is 0 (no raw frames at all), the existing `Math.max(..., 1)` still ensures at least 1 iteration, and the compositor will throw at the "could not resolve a source frame" check. This is correct — a recording with zero frames is an error.

**Edge case guard 2:** If `effectiveFrames.length` is 0 but `effectiveFrameFiles.length` > 0, this means the timeline has no frame data but raw frames exist. This is a malformed timeline — the `Math.max(..., 1)` produces 1 frame, which is acceptable as a degenerate case. The prior behavior (`Math.max` of both) would also produce the file count, which would create a video without timing data — arguably worse.

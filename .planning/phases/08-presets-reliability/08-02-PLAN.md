  <name>Task 1: Add signal handling to compose() function</name>
  <files>packages/core/src/compositor.ts</files>
  <action>
    Add SIGINT/SIGTERM signal handling to the `compose()` function, following the existing pattern from `recorder.ts` lines 176-226.

    Implementation:
    1. At the start of `compose()`, before the tempDirectory creation:
       - Declare `const signals: NodeJS.Signals[] = ["SIGINT", "SIGTERM"]`
       - Declare `let interrupted = false`
       - Create `handleSignal` that sets `interrupted = true`
       - Register with `process.once(signal, handleSignal)` for each signal

    2. In the `try` block, check `interrupted` after each major operation:
       - After `decodeRawFrames()` — check before starting the frame compositing loop
       - Inside the frame loop — check every ~100 frames (use `frameIndex % 100 === 0`) to avoid per-frame overhead
       - After the frame loop, before `encodeFrames()` — check again

    3. When interrupted, throw `new Error("Compositing interrupted by signal")` so the `finally` block still runs for cleanup.

    4. In the `finally` block (which already cleans up tempDirectory), also:
       - Remove signal handlers with `process.off(signal, handleSignal)`
       - If interrupted, attempt to remove partial output file: `await rm(outputPath, { force: true }).catch(() => {})` — only if the file was being written (check `format !== "gif"` for the intermediate path)

    IMPORTANT: The recorder already handles signals when `record()` calls `compose()` internally (recorder.ts line 332). The recorder catches the signal first, cleans up, and never reaches compose's signal handler. This compositor signal handling is for standalone `tuireel composite` only. Use `process.once` (not `process.on`) to avoid double-handler issues.

    Do NOT use `process.exit()` in the signal handler — let the error propagate and the event loop drain naturally (matching existing recorder pattern).
  </action>

    - grep confirms: `process.off.*handleSignal` appears in compositor.ts
  </verify>

    ```

    ```

    New error format:
    ```
    ffmpeg failed to {operationDescription} (exit code: {exitCode}).
    Command: {ffmpegPath} {args.join(" ")}
    {stderr}
    ```

    Implementation in the catch block of `runFfmpeg()`:
    1. Extract exit code: `const exitCode = (error as { code?: number }).code ?? "unknown"`
    2. Build command string: `const command = [ffmpegPath, ...args].join(" ")`
    3. Format error: include exit code, command, and stderr (from existing `getErrorStderr()`)
    4. Do NOT include the original `error.message` — it typically repeats the stderr content. The command + exit code + stderr is more useful.
    5. Preserve the `cause` chain: `throw new Error(message, { cause: error })`

    Also add `{ cause: error }` to make the original error available for debugging.

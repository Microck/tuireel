  <behavior>
    Preset resolution merges a named preset's defaults under user config.

    Merge priority: User Config > Preset Defaults > Schema Defaults (Zod .default()).

    Cases:
    - resolvePreset({ preset: "polished", steps: [...] }) -> { theme: "catppuccin", sound: { effects: { click: 1, key: 1 } }, cursor: { visible: true }, hud: { visible: true }, steps: [...] }
    - resolvePreset({ preset: "polished", theme: "tokyo-night", steps: [...] }) -> { theme: "tokyo-night", sound: { effects: { click: 1, key: 1 } }, ... } (user theme wins)
    - resolvePreset({ preset: "polished", sound: { effects: { key: 2 } }, steps: [...] }) -> { sound: { effects: { key: 2 } }, ... } (user sound replaces preset sound entirely — user explicitly set sound)
    - resolvePreset({ preset: "silent", steps: [...] }) -> { cursor: { visible: false }, hud: { visible: false }, steps: [...] } (no theme, no sound)
    - resolvePreset({ preset: "minimal", steps: [...] }) -> { theme: "tokyo-night", cursor: { visible: true }, hud: { visible: false }, steps: [...] }
    - resolvePreset({ steps: [...] }) -> { steps: [...] } (no preset, passthrough)
    - resolvePreset({ preset: "nonexistent", steps: [...] }) -> throws (unknown preset)

    Sound merge rule: If the user explicitly provides a `sound` field (even partial), it REPLACES the preset's sound entirely. This avoids the confusing "partial sound merge" pitfall. If the user doesn't set sound at all, preset sound applies.

    Schema additions to baseConfigFields:
    - `preset`: z.enum(PRESET_NAMES).optional()
    - `cursor`: z.object({ visible: z.boolean().optional() }).optional()
    - `hud`: z.object({ visible: z.boolean().optional() }).optional()
    - `defaultWaitTimeout`: z.number().positive().optional() (no .default() — absence means no global timeout)
  </behavior>
  <implementation>
    1. Create `packages/core/src/presets/built-in.ts`:
       - Export `PRESET_NAMES = ["polished", "minimal", "demo", "silent"] as const`
       - Export `type PresetName = typeof PRESET_NAMES[number]`
       - Define `PresetConfig` type as `Partial<Omit<TuireelConfig, "steps" | "output" | "$schema" | "format">>` — presets set presentation, not output behavior
       - Export `BUILT_IN_PRESETS: Record<PresetName, PresetConfig>` with:
         - polished: { theme: "catppuccin", sound: { effects: { click: 1, key: 1 } }, cursor: { visible: true }, hud: { visible: true } }
         - minimal: { theme: "tokyo-night", cursor: { visible: true }, hud: { visible: false } }
         - demo: { theme: "dracula", sound: { effects: { click: 1, key: 1 } }, cursor: { visible: true }, hud: { visible: true } }
         - silent: { cursor: { visible: false }, hud: { visible: false } }

    2. Modify `packages/core/src/config/schema.ts`:
       - Import `PRESET_NAMES` from `../presets/built-in.js`
       - Add to baseConfigFields:
         - `preset: z.enum(PRESET_NAMES).optional()`
         - `cursor: z.object({ visible: z.boolean().optional() }).optional()`
         - `hud: z.object({ visible: z.boolean().optional() }).optional()`
         - `defaultWaitTimeout: z.number().positive().optional()`
       - Export the cursor and hud schema types

    3. Create `packages/core/src/presets/resolve.ts`:
       - Export function `resolvePreset(rawConfig: Record<string, unknown>): Record<string, unknown>`
       - If no `preset` field, return rawConfig unchanged
       - Look up preset in BUILT_IN_PRESETS, throw if not found
       - Strip `preset` key from output (it's resolved, not passed through)
       - Spread: `{ ...presetDefaults, ...userOverrides }` where userOverrides is rawConfig minus the `preset` key
       - Sound handling: if user has `sound` property (even undefined), user wins entirely. If user has no `sound` key, preset sound applies.

    4. Create `packages/core/src/presets/index.ts`:
       - Re-export from built-in.ts and resolve.ts

    5. Test file `packages/core/test/presets/resolve.test.ts` covering all cases above.
  </implementation>

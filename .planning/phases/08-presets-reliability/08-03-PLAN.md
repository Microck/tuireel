  <name>Task 1: Wire preset resolution into config loader</name>
  <files>packages/core/src/config/loader.ts</files>
  <action>
    Insert preset resolution into the config loading pipeline so that preset defaults are applied BEFORE Zod validation.

    In `loadConfigFromString()` (line 217):
    1. Import `resolvePreset` from `../presets/resolve.js`
    2. After `parseConfig()` (which returns raw unknown), call `resolvePreset()` on the parsed config BEFORE `parseInputConfig()`
    3. For multi-video configs (has `videos` key): do NOT apply presets at the top level. Multi-video configs use `defaults` for shared config — presets are for single-video configs only. This keeps things simple and avoids confusing interactions.

    Implementation:
    ```
    const parsedConfig = parseConfig(rawConfig);
    const presetResolved = resolvePreset(parsedConfig); // NEW - applies preset before validation
    const stepTypeIssues = findInvalidStepTypeIssues(presetResolved);
    ...
    const inputConfig = parseInputConfig(presetResolved);
    ```

    Also wire preset resolution into `parseSingleConfig()` (line 170) for the `validateConfig()` path:
    ```
    const parsedConfig = parseConfig(rawConfig);
    const presetResolved = resolvePreset(parsedConfig);
    const stepTypeIssues = findInvalidStepTypeIssues(presetResolved);
    const result = configSchema.safeParse(presetResolved);
    ```

    The `preset` field will be stripped by `resolvePreset()` (it spreads preset defaults and removes the `preset` key). After resolution, the config looks like a normal config with all fields filled in — Zod validates it as usual.

    NOTE: The `preset` field is in the schema (for JSON Schema generation and IDE autocomplete) but `resolvePreset` strips it before Zod parse. The Zod schema has `preset` as optional, so `.parse()` will just ignore it if present (Zod strips unknown fields by default with `.object()`). But cleaner to strip it in resolvePreset.
  </action>

  <action>
    **Init command (init.ts):**
    Add an interactive preset selection prompt to `tuireel init` using Node.js built-in `readline`.

    1. Import `createInterface` from `node:readline`
    2. Import `PRESET_NAMES` from `@tuireel/core` (add to the core package exports if needed)
    3. Create `promptPreset()` async function that:
       - Displays numbered preset choices with descriptions:
         ```
         Choose a preset (optional):
           1. polished  - catppuccin + sound + cursor + HUD
           2. minimal   - tokyo-night + cursor only
           3. demo      - dracula + sound + cursor + HUD
           4. silent    - no overlays, no sound
           5. none      - start from scratch
         ```
       - Reads a single line of input via readline
       - Returns the preset name string, or undefined for "none"/invalid/empty
    4. Call `promptPreset()` in the init action, before generating the config
    5. If a preset is selected, add `"preset": "{name}"` to the generated starter config (add it after `"$schema"` line)
    6. If stdin is not a TTY (piped input), skip the prompt and use no preset (for CI/scripting)

    Check for TTY: `process.stdin.isTTY` — if falsy, skip prompt.

    **Recorder cursor/hud passthrough (recorder.ts):**
    In the `record()` function, when calling `compose()` (around line 332), pass cursor and hud config from the resolved config:

    Currently:
    ```typescript
    await compose(artifacts.rawVideoPath, timeline.toJSON(), config.output, {
      format: config.format,
      sound: config.sound,
    });
    ```

    Change to:
    ```typescript
    await compose(artifacts.rawVideoPath, timeline.toJSON(), config.output, {
      format: config.format,
      sound: config.sound,
      cursorConfig: config.cursor ? { visible: config.cursor.visible ?? true } : undefined,
      hudConfig: config.hud ? { visible: config.hud.visible ?? true } : undefined,
    });
    ```

    This ensures that preset-defined cursor/hud visibility flows through to compositing. The `ComposeOptions` interface already accepts `cursorConfig` and `hudConfig` — no interface changes needed.

    NOTE: The `CursorConfig` and `HudConfig` types in `overlay/types.ts` may have additional fields beyond `visible`. Only map the `visible` field from the config schema. Other fields (size, color, position) remain at their defaults.
